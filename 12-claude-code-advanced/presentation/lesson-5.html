Headless режим и платформы Claude Code

45 минут

Headless режим

Аналогия

Обычный Claude Code - это как разговор по телефону (диалог).
Headless - это как отправить SMS с заданием и получить ответ (одна команда).

Базовое использование:

Terminal
# Просто задать вопрос
$ claude -p "Что делает этот проект?"

# Запустить и починить тесты
$ claude -p "Запусти тесты и исправь ошибки" --allowedTools "Bash,Read,Edit"

# Получить результат в JSON
$ claude -p "Опиши проект" --output-format json

Мощные примеры:

Pipe - передать данные через канал:

# Анализ логов
$ tail -f app.log | claude -p "Уведоми меня в Slack если увидишь аномалии"

# Code review PR
$ gh pr diff 123 | claude -p "Проверь код на уязвимости"

# Список изменённых файлов -> ревью
$ git diff main --name-only | claude -p "Проверь эти файлы на security issues"

Продолжение разговора:

# Первая команда
$ claude -p "Проверь проект на проблемы"

# Продолжить тот же разговор
$ claude -p "Теперь исправь найденные проблемы" --continue

Headless в CI/CD

GitHub Actions:

# .github/workflows/claude-review.yml
name: Claude Code Review
on: [pull_request]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Claude Review
        run: claude -p "Check for security issues and post a review"
Идеально для

Code review: Каждый PR автоматически проверяется Claude на безопасность и качество.
Changelog: При каждом релизе Claude пишет список изменений по коммитам.
Автокоммиты: Claude исправляет линтинг-ошибки и коммитит без вашего участия.
Мониторинг: Claude анализирует логи и отправляет алерты в Slack.

Agent Teams - команда агентов

Экспериментальная фича

Agent Teams пока экспериментальные. Включаются через настройку в settings.json:

{ "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" } }
Аналогия

Субагент - это как отправить курьера с заданием. Он уходит, делает работу и возвращает результат. Курьеры между собой не общаются.

Agent Team - это как собрать рабочую группу. Есть руководитель (Lead), есть участники (Teammates). У них общий список задач и они могут общаться между собой напрямую - обсуждать, спорить, делиться находками.

Архитектура команды:

КомпонентРоль
Lead (руководитель)Ваша основная сессия. Создаёт команду, раздаёт задачи, собирает результаты
Teammates (участники)Отдельные Claude сессии. Каждый работает самостоятельно
Task List (задачи)Общий список задач. Участники сами берут свободные задачи
Mailbox (почта)Система сообщений. Участники пишут друг другу напрямую

Субагенты vs Agent Teams:

СубагентыAgent Teams
КонтекстСвой, результат возвращается вамСвой, полностью независимый
ОбщениеТолько с основным агентомМежду собой напрямую
КоординацияClaude управляет всемОбщий список задач + самоорганизация
Лучше дляБыстрых задач, где нужен только результатСложных задач, где нужно обсуждение
ТокеныМеньше (результат суммируется)Больше (каждый - отдельная сессия)

Как запустить команду:

Просто опишите задачу и попросите создать команду:

Claude Code
> Создай команду из 3 агентов для ревью PR #142:
- Один проверяет безопасность
- Второй проверяет производительность
- Третий проверяет покрытие тестами
Пусть обсудят и дадут общие рекомендации.

Режимы отображения:

РежимКак работаетНавигация
In-process (по умолчанию)Все в одном терминалеShift+Down - переключиться между участниками
Split panesКаждый в своей панели (через tmux/iTerm2)Кликнуть в панель участника

Пример: Дебаг с конкурирующими гипотезами

Когда непонятно в чём баг, можно запустить несколько агентов с разными теориями. Они будут спорить между собой и опровергать друг друга:

Claude Code
> Пользователи жалуются что приложение закрывается после первого сообщения.
Создай команду из 5 агентов. Каждый исследует свою гипотезу.
Пусть общаются между собой и опровергают теории друг друга.
Обнови документ с выводами команды.
Когда использовать Agent Teams

Хорошо подходит для: параллельный code review, исследование с разных сторон, разработка независимых модулей, дебаг с несколькими гипотезами.

Не стоит для: последовательных задач, изменений в одном файле, простых задач (субагенты справятся лучше и дешевле).

Управление командой:

  • Общение напрямую: Shift+Down чтобы перейти к участнику и написать ему
  • Одобрение планов: можно потребовать чтобы участники показывали план перед реализацией
  • Хуки: TeammateIdle - когда участник простаивает, TaskCompleted - когда задача завершена
  • Завершение: попросите Lead "Clean up the team" для корректного завершения

Плагины и Маркетплейс

Аналогия

Плагин - это как приложение из App Store. Кто-то собрал полезный набор (Skills, Hooks, агенты, MCP серверы) в одну коробку, и вы устанавливаете его одной командой. Маркетплейс - это сам магазин, где хранятся плагины.

Зачем плагины?

Без плагинов вы настраиваете Skills, Hooks и агенты вручную в каждом проекте. С плагинами:

  • Один раз собрали - используете везде (все проекты)
  • Делитесь с командой (через маркетплейс)
  • Версионирование (обновления как у обычных пакетов)
  • Нет конфликтов - навыки автоматически получают префикс: /my-plugin:review

Структура плагина:

my-plugin/
├── .claude-plugin/
│   └── plugin.json        # Имя, описание, версия
├── skills/                 # Skills плагина
│   └── review/SKILL.md
├── agents/                 # Агенты плагина
├── hooks/                  # Хуки плагина
│   └── hooks.json
├── .mcp.json              # MCP серверы плагина
└── settings.json          # Настройки по умолчанию

Установка и использование:

Claude Code
# Добавить маркетплейс
> /plugin marketplace add acme-corp/claude-plugins

# Установить плагин из маркетплейса
> /plugin install review-plugin@acme-plugins

# Использовать (с префиксом плагина)
> /review-plugin:review

# Тестировать локальный плагин
$ claude --plugin-dir ./my-plugin

Свой плагин vs обычные файлы:

Обычные (.claude/)Плагин
Имена Skills/review/my-plugin:review
ДоступностьТолько этот проектЛюбой проект
ПоделитьсяКопировать вручнуюУстановить одной командой
Для чегоЛичные настройки, экспериментыШаринг с командой, сообществом
Совет

Начните с обычных файлов в .claude/. Когда навыки и хуки станут стабильными - упакуйте их в плагин для переиспользования. Есть даже официальный маркетплейс Anthropic, куда можно отправить свой плагин!

Git Worktrees - параллельная работа

Что это?

Git worktree позволяет иметь несколько рабочих копий одного репозитория. Каждая на своей ветке. Идеально для параллельной работы нескольких Claude сессий!

# Терминал 1: Основная работа
$ cd project && claude

# Терминал 2: Фича в изолированной копии
$ cd project && claude -w feature/auth

# Терминал 3: Багфикс в другой копии
$ cd project && claude -w fix/memory-leak

# Все работают параллельно, не мешая друг другу!

Практика

Задание 1: Headless проверка

$ claude -p "Найди все TODO комментарии в проекте и создай список"

Задание 2: Pipe данных в Claude

$ git log --oneline -20 | claude -p "Напиши changelog на основе этих коммитов"

Задание 3: Параллельные worktrees

# Откройте 2 терминала и запустите Claude в каждом
$ claude -w feature/my-feature
